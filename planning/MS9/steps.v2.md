# MS9 — PyO3 binding (`crates/python`, maturin) — STEP plan (iteration 2)

> **Milestone:** MS9 (the LAST v0.1 milestone) — expose `validate` + `describe` to
> Python via a thin PyO3 binding that **mirrors, never reimplements** the
> `hdx-core` verbs (spec §10; architecture §2).
> **Source contract:** `spec/HDX_SPEC.md`. **Planned against:** `architecture.md`
> + `planning/milestones.md` (MS9 section).
> **Scope cut (hard):** ONLY `validate` + `describe`. No `regrid`/`clip`/`reduce`,
> no new `hdx-core` domain/manifest field, no inert-violating field anywhere. The
> binding adds **zero** contract logic — it calls `hdx-core` and returns the same
> JSON-shaped results to Python.

## Verified starting state (read from the repo before planning)

- **Verbs to mirror** (`hdx-core` public API, exact signatures):
  - `hdx_core::describe::describe_json(path: impl AsRef<Path>) -> Result<String, DescribeError>`

  - `hdx_core::validate::validate_json(path: impl AsRef<Path>) -> Result<String, ValidateError>`

    `ValidationReport::conformant() -> bool`.

  `DescribeError`/`ValidateError` each have variants

  `Discovery(CoreError)`, `Serialize { detail }`. The **§0 hard cut** surfaces as
  `Manifest(CoreError::UnknownFormatVersion { found })` — it is an `Err`, **never**
  softened into a `conformant:false` report (verified in `validate`/`describe`
  entry-order docs and in `crates/core/src/error.rs`).
- **Wrong-version fixture:** `conformance/invalid/wrong-format-version/` (manifest

  pinned invalid at `conformance/invalid/missing-root-rollup/` (reports
  `conformant:false`, not an error — verified).
- **Fixtures-gitignored policy (IN EFFECT):** `git ls-files conformance/valid/*
  conformance/invalid/*` returns **0** files; `.gitignore` ignores
  `conformance/valid/**` + `conformance/invalid/**`; goldens are tracked under
  `conformance/goldens/`. Fixtures are regenerated by
  `PYTHON=python3.12 conformance/generator/regenerate.sh` (idempotent venv at
  `conformance/generator/.venv`, pinned with `.lock.sha256`).
- **Toolchain present on host:** `/usr/local/bin/python3.12`,
  `/opt/homebrew/bin/maturin`. Host is **Darwin arm64** — the macOS PyO3 link
  gotcha (HIGH-1/HIGH-2) is real and must be designed around from the first commit.
- **Workspace:** root `Cargo.toml` has `members = ["crates/*"]`, so a new
  `crates/python` is auto-included — `cargo build/test/clippy --all-targets`
  immediately covers it. Only `crates/core` exists today; `crates/python` is new.

## Ordering rationale

MS9 is the final, deliberately minimal milestone: a thin PyO3 wrapper. Three
dependency-sequential steps, each independently committable and green:

1. **S1 — Stand up the green PyO3 crate skeleton (toolchain wiring + import
   proof).** Everything downstream depends on a crate that *links* on macOS under
   the whole-workspace gate. The single highest-risk thing in MS9 is the macOS
   PyO3 link configuration (the prior critique's two HIGH defects), so it is
   isolated and proven **first**, with the smallest possible surface (one trivial
   `__core_version()` function). This pins `crate-type = ["cdylib", "rlib"]` +
   the **optional, non-default** `extension-module` feature, so `cargo
   build/test/clippy --all-targets -- -D warnings` stay green on Darwin/arm64
   (the rlib test target links **without** `extension-module`), while `maturin
   develop`/`build` enable the feature for the shipped wheel. It proves the link
   end-to-end with a real `python3.12 -c "import hdx; hdx.__core_version()"`
   against a wheel artifact. It also records the macOS link gotcha as an
   `architecture.md` amendment (the only architecture edit MS9 performs, done by
   the implementer at S1 — this plan does not edit architecture).

2. **S2 — The two mirrored verbs + error→exception mapping (Rust side, unit
   proven).** With a linkable crate, add the actual `validate`/`describe` wrappers
   that call `hdx-core::*_json`, parse the returned JSON string into a Python
   object (so callers get a `dict`, not a string), and map `DescribeError`/
   `ValidateError` → Python exceptions — crucially mapping the §0 hard cut
   (`Manifest(UnknownFormatVersion)`) to a clear, dedicated exception type so it
   is **never** softened into a `conformant:false` result. The error-mapping
   helper is proven by `#[cfg(test)]` **Rust** unit tests over the `rlib` target
   (buildable because S1 set `["cdylib","rlib"]` + gated the feature) — the only
   Rust-level proof of the §0 mapping before the Python tests in S3. No rule
   logic, no wire-shape re-derivation: the binding reuses the exact `*_json`
   strings, so the Python output matches the committed schemas by construction.

3. **S3 — Python integration tests over regenerated fixtures + README.** Last,
   prove the binding end-to-end from Python against **regenerated** fixtures
   (regenerate-first prerequisite), in an **isolated, dedicated binding venv**
   (`crates/python/.venv`, never the generator's `conformance/generator/.venv`):
   `describe`/`validate` over the valid fixture return the expected structures
   (dicts whose keys match the schemas), `validate` over a regenerated invalid
   returns `conformant: false`, and the wrong-format-version fixture **raises the
   hard-cut exception** (§0 preserved end-to-end to Python). A short
   `crates/python/README.md` documents maturin build + usage + the
   mirrors-hdx-core (no contract logic) note + the regenerate-first test
   prerequisite + the isolated-venv path. This is the milestone's reviewable
   outcome.

Dependencies: **S1 → S2 → S3.** S1 must precede S2 because the verbs need a crate
that links; S2 must precede S3 because the Python tests import the functions S2
adds. Each step ends green for the whole workspace.

## Scope-guard statement

Every step stays strictly inside MS9: the binding exposes **only** `validate` +
`describe`, each a thin pass-through to `hdx-core`'s `*_json` verbs. **No** step
introduces `regrid`/`clip`/`reduce`, adds a new `hdx-core` domain or manifest
field, re-derives the §14 rule logic or the JSON wire shape in the binding, or
adds any inert-violating field (transform/role/semantic/provenance) to any type.
`hdx-core` is **frozen** for MS9 — no reader, rule, domain-type, or manifest-floor
change; MS9 adds only `crates/python`, its tests, its README, and one
`architecture.md` amendment recording the macOS link setup. No step does a later
milestone's work — MS9 is the last milestone, so "later milestone" is moot, but
the binding never reaches beyond mirroring the two existing verbs. The §0 hard
version cut is preserved end-to-end: the wrong-format-version fixture raises a
hard-cut exception (surfacing `UnknownFormatVersion`), never a softened
`conformant:false`. The fixtures-gitignored / regenerate-first policy is upheld:
no fixture bytes are committed, tests regenerate first, and a clean regenerate
leaves all tracked `conformance/goldens/` intact (the binding reads fixtures, adds
none).

## MS9-S1 — Green PyO3 crate skeleton: macOS-safe toolchain wiring + import proof

**id:** MS9-S1

**Intent.** Stand up `crates/python` as a PyO3 abi3 extension crate that **links
and tests green on Darwin/arm64 under the whole-workspace gate** — solving the two
HIGH macOS-link defects up front with the smallest possible surface. Ship exactly
one trivial function, `__core_version()`, whose only job is to prove the
extension imports and links against `hdx-core` from Python. This is independently
committable (a buildable, importable, do-nothing-useful extension) and leaves the
repo green: `cargo build`, `cargo test`, and `cargo clippy --all-targets --
-D warnings` pass for the whole workspace (the new crate's `rlib` test target
links **without** `extension-module`), and `maturin develop`/`build` produce an
importable abi3 wheel (maturin enables `extension-module` for the wheel only).

**Why this is the canonical macOS wiring (folds HIGH-1, HIGH-2, LOW-1):**
- `[lib] crate-type = ["cdylib", "rlib"]` (HIGH-2) — the `cdylib` is the Python
  extension; the `rlib` gives `cargo test`/`clippy --all-targets` a normal Rust
  library target to build `#[cfg(test)]` unit tests against (S2's error-mapping
  tests need this). A `cdylib`-only crate would have **no** buildable unit-test
  target, so S1's and S2's Rust unit tests would be unrealizable.
- `extension-module` is declared as an **optional, non-default** feature (HIGH-1):
  `[features] extension-module = ["pyo3/extension-module"]`. `pyo3` is added with
  `default-features = false` so the feature is OFF for `cargo build/test/clippy`.
  With the feature OFF the test executable links normally on macOS; with it ON
  (maturin's default) the wheel's `cdylib` leaves the `_Py*` symbols undefined for
  resolution at import time (the correct macOS/Windows behavior). Enabling
  `extension-module` unconditionally is the canonical PyO3 gotcha that breaks
  linking of every non-cdylib target on macOS — this plan never enables it
  unconditionally; `cargo` runs it OFF, `maturin` turns it ON via
  `[tool.maturin] features`.

**Changes.**
- `crates/python/Cargo.toml` (new): package `hdx-python` (the importable Python
  module is named `hdx`), `edition = "2024"`, version mirrored to the workspace
  bump convention. Dependencies: `pyo3` (with `abi3-py312` so the wheel is a
  single abi3 wheel against the python3.12 ABI floor; `default-features = false`
  to keep `extension-module` OUT of the default set), and
  `hdx-core = { path = "../core" }`. `[lib] name = "hdx"`, `crate-type =
  ["cdylib", "rlib"]`. `[features] extension-module = ["pyo3/extension-module"]`
  (optional, non-default). Each non-obvious choice carries a `#`-comment
  explaining the macOS link rationale (mirroring the heavily-commented
  `crates/core/Cargo.toml` convention).
- `crates/python/src/lib.rs` (new): the `#[pymodule] fn hdx(...)` entry point
  registering exactly one function — `#[pyfunction] fn __core_version() -> &'static str`
  returning a compile-time constant (e.g. `env!("CARGO_PKG_VERSION")`) so the
  import/link is exercised with zero IO and zero contract logic. Module `//!` doc:
  one line stating this crate is a thin PyO3 mirror of `hdx-core` (no contract
  logic), and that `extension-module` is an optional feature maturin enables.
  `tracing` only if any diagnostic is needed (none expected here); never
  `println!`.
- `crates/python/pyproject.toml` (new): `[build-system] requires = ["maturin>=1,<2"]`,
  `build-backend = "maturin"`; `[tool.maturin] features = ["extension-module"]`
  (so the wheel build enables the optional feature) and the abi3 settings; project
  metadata naming the module `hdx`, `requires-python = ">=3.12"`.
- Root `.gitignore` (append): ignore the binding's build/test venv and wheel
  outputs — `crates/python/.venv/`, `crates/python/**/__pycache__/`,
  `target/wheels/` (the maturin wheel output dir) — so no build artifact or wheel
  is ever committed (mirrors the conformance generator's venv-ignore discipline).
- No change to `crates/core` or `src/main.rs` (hdx-core frozen). `crates/python`
  is picked up automatically by `members = ["crates/*"]`.

**test_plan.**
- Rust: a `#[cfg(test)]` unit test in `crates/python/src/lib.rs` (built against
  the **rlib** target, HIGH-2) asserting `__core_version()` returns a non-empty
  `&str` — it compiles and runs WITHOUT `extension-module` (the macOS-link proof
  at the Rust level, HIGH-1).
- Build gate (asserted in acceptance): `cargo build`, `cargo test`, and
  `cargo clippy --all-targets -- -D warnings` are run for the **whole workspace**
  and pass; additionally `cargo test -p hdx-python` and
  `cargo clippy --all-targets -p hdx-python -- -D warnings` pass on macOS (the
  explicit HIGH-1 fold — the non-cdylib targets link with the feature OFF).
- Maturin/import proof (the LOW-1 fold, concrete not soft): build the wheel with
  maturin into a throwaway python3.12 venv (`maturin develop` from
  `crates/python` into a python3.12 venv, OR `maturin build` then assert a
  `target/wheels/*-abi3-*.whl` artifact exists), then run
  `python3.12 -c "import hdx; hdx.__core_version()"` and assert it exits 0. This

- `crate-type = ["cdylib", "rlib"]` is set so the unit-test target exists (HIGH-2).
- `extension-module` is an optional, non-default feature gated behind
  `pyo3/extension-module`; `cargo` never enables it; `[tool.maturin] features`
  enables it for the wheel only.
- A `*-abi3-*.whl` artifact is produced under `target/wheels/` (or via

- No fixture data committed; no `hdx-core`/`main.rs` change.
- Commit via `./scripts/bump-version.sh patch`, stage `Cargo.toml` (+ new files),

## MS9-S2 — Mirror `validate` + `describe`; map `hdx-core` errors to Python exceptions (§0 hard cut preserved)

**id:** MS9-S2

**Intent.** Add the two mirrored verbs and the error→exception mapping — the
substance of MS9 — while adding **zero** contract logic. `hdx.validate(path)` and
`hdx.describe(path)` each call the corresponding `hdx-core` `*_json` verb, parse
the returned JSON string into a Python object (so callers get a `dict`), and on
error map the typed `DescribeError`/`ValidateError` to Python exceptions. The §0
hard cut (`Manifest(CoreError::UnknownFormatVersion)`) maps to a **dedicated,
clearly-named** Python exception (surfacing `UnknownFormatVersion`), never
softened into a `conformant:false` result. Because the binding reuses the exact
`*_json` strings, the Python `dict` matches `schemas/describe.schema.json` /
`schemas/validate.schema.json` by construction — no wire shape is re-derived. This
is independently committable (the functions compile, the Rust error-mapping unit
tests pass) and leaves the repo green.

**Mirror discipline (folds the STEP-2 "mirror, never reimplement" mandate):**
- The binding calls **only** `hdx_core::describe::describe_json` and
  `hdx_core::validate::validate_json` — no §14 rule, no contract logic, no
  re-derived wire shape lives here.
- To return a `dict` (not a raw string), the binding parses the `*_json` string
  with a JSON→Python conversion (e.g. import Python's `json` module via `pyo3` and
  call `json.loads`, or convert a `serde_json::Value` → `PyObject`). The chosen
  mechanism only **transports** the already-produced JSON; it does not reshape or
  re-derive it. (If a `dict` is impractical, return the JSON string and document
  that callers `json.loads` it — but the default deliverable is a `dict`.)
- A note in the module doc states the binding mirrors `hdx-core` and adds no
  contract logic.

**Error mapping (folds the §0 end-to-end mandate):**
- Define Python exception types via PyO3 (`create_exception!`), with a small base
  (e.g. `HdxError`) and a dedicated subclass for the hard cut
  (e.g. `UnknownFormatVersionError`).
- A single mapping helper turns each `DescribeError`/`ValidateError` variant into
  the right exception: `Manifest(CoreError::UnknownFormatVersion { found })` →
  `UnknownFormatVersionError` (message surfacing the offending version);
  `ManifestUnreadable`, other `Manifest(_)`, `Discovery(_)`, `Serialize` →
  `HdxError` (or specific subclasses) carrying the `Display` message. No `Err`
  is ever turned into a `conformant:false` value — `validate` only returns a
  `dict` on `Ok`; a `conformant:false` is a normal `Ok` report `dict`, distinct
  from any raised exception.

**Changes.**
- `crates/python/src/lib.rs` (modify): add `#[pyfunction] fn validate(path: &str) -> PyResult<...>`
  and `#[pyfunction] fn describe(path: &str) -> PyResult<...>`, each calling the
  matching `hdx-core` `*_json`, parsing the JSON into a Python object, and mapping
  errors via the helper. Register both (plus `__core_version`) in the
  `#[pymodule]`. Add `create_exception!`-defined exception types and register
  them on the module.
- `crates/python/Cargo.toml` (modify if needed): add only what the JSON→Python
  transport needs. Prefer **no new dependency** — use `serde_json::Value` (already
  in `hdx-core`'s closure; add `serde_json` directly to `crates/python` if needed)
  + a manual `Value → PyObject` conversion, OR call Python's stdlib `json.loads`
  via `pyo3` (no Rust dep at all). If a tiny helper crate (`pythonize`) is added,
  pin its major and `#`-comment why; keep deps minimal.
- No `hdx-core`/`main.rs` change.

**test_plan.**
- Rust `#[cfg(test)]` unit tests (built against the **rlib** target, link green
  on macOS because `extension-module` is OFF — relies on S1's `["cdylib","rlib"]`
  + optional feature): exercise the error-mapping helper —
  - a `ValidateError::Manifest(CoreError::UnknownFormatVersion { found: "0.2" })`
    maps to the dedicated hard-cut exception type (the §0 mapping proof at the
    Rust level, before S3's Python tests);
  - a `DescribeError::ManifestUnreadable { .. }` maps to the general `HdxError`;
  - a generic `Discovery(_)` maps to `HdxError`.
    (These assert exception **type** selection — the load-bearing §0 distinction
    — without needing a Python interpreter, and are buildable only because S1 made
    the crate an `rlib` too.)
- Build gate: `cargo build`, `cargo test`, `cargo clippy --all-targets --
  -D warnings` pass for the whole workspace; `cargo test -p hdx-python` passes on
  macOS.

**acceptance.**
- `hdx.validate` / `hdx.describe` exist as `#[pyfunction]`s registered on the
  `hdx` module; each calls only `hdx-core`'s `*_json` verb (no rule logic, no
  wire-shape re-derivation) — verifiable by review of `lib.rs`.
- The §0 hard cut maps to a dedicated `UnknownFormatVersion`-style Python
  exception; a Rust unit test asserts the mapping from
  `Manifest(CoreError::UnknownFormatVersion)` selects that exception type and
  **not** a report value (the hard cut is never softened to `conformant:false`).
- Other `DescribeError`/`ValidateError` variants map to a clear `HdxError`-family
  exception carrying the `Display` message; unit-tested.
- The returned Python value is a `dict` produced by transporting the `*_json`
  output unchanged (or, if string, documented for `json.loads`) — the keys are
  exactly the schema keys (`manifest/basins/fields/grids/time_extents/delineations`
  for describe; `checks/conformant` for validate) because the string is reused
  verbatim.
- `cargo build`, `cargo test`, `cargo clippy --all-targets -- -D warnings` pass
  for the whole workspace; `cargo test -p hdx-python` passes on macOS.
- No new `hdx-core` field/rule; no inert violation; no fixture committed.
- Commit via `./scripts/bump-version.sh patch`, stage `Cargo.toml`, conventional
  commit, `git tag v<version>`.

**spec_refs.** §0 (hard cut preserved through the binding — wrong version is an
exception, never a softened report); §10 ("PyO3 mirrors them"; the binding wraps
the verbs, adds no contract); §1 (inert/agnostic — no new field); §14 (the
binding exposes MS6's enforcement unchanged via the reused `validate_json`);
R4 (same stable JSON shape).

**commit_message.** `feat(python): mirror validate/describe verbs with error-to-exception mapping`

## MS9-S3 — Python integration tests over regenerated fixtures (regenerate-first, isolated venv) + README

**id:** MS9-S3

**Intent.** Prove the binding end-to-end from Python and document it — the
milestone's reviewable outcome. Python tests, run from an **isolated, dedicated
binding venv** with `hdx` installed via `maturin develop`, exercise the verbs over
**regenerated** fixtures: `describe`/`validate` over the valid fixture return the
expected structures (dicts whose keys match the schemas), `validate` over a
regenerated invalid returns `conformant: false`, and the wrong-format-version
fixture **raises the hard-cut exception** (§0 preserved all the way to Python). A
short `crates/python/README.md` documents the maturin build, usage, the
mirrors-hdx-core (no contract logic) note, the **regenerate-first** test
prerequisite, and the isolated-venv path. This is independently committable and
leaves the repo green; the Python test layer is dev-only tooling (not shipped).

**Fixtures-gitignored / regenerate-first discipline (folds the STEP-2 mandate +
MED-1):**
- The Python tests read fixtures from `conformance/valid/minimal/`,
  `conformance/invalid/missing-root-rollup/` (or another regenerated invalid), and
  `conformance/invalid/wrong-format-version/`. Because those trees are git-ignored
  (verified: 0 tracked files), the test runner script **regenerates first**:
  `PYTHON=python3.12 conformance/generator/regenerate.sh` BEFORE any test reads a
  fixture. No fixture bytes are committed by this step.
- **Isolated venvs (MED-1).** The binding test venv is a **dedicated**
  `crates/python/.venv` (ignored by S1's `.gitignore` addition), created and
  populated by `maturin develop` into THAT venv; the generator's
  `conformance/generator/.venv` is left **untouched**. The regenerate step runs
  under its own generator venv (its `regenerate.sh` manages it); the test step
  runs under `crates/python/.venv`. The runner script states both exact venv
  paths so the run is deterministic and the two interpreters never collide.
- After a clean regenerate, all tracked goldens under `conformance/goldens/`
  remain intact (the binding reads fixtures, writes none; no golden lives under
  the gitignored trees — verified policy). The runner does not write into
  `conformance/valid/**` or `conformance/invalid/**`.

**Changes.**
- `crates/python/tests/` (new): a `pytest` test module (e.g.
  `test_mirror.py`) with three tests:
  1. `describe(valid)` and `validate(valid)` return dicts whose top-level keys
     match the schemas (`manifest/basins/fields/grids/time_extents/delineations`;
     `checks/conformant`), and `validate(valid)["conformant"] is True`.
  2. `validate(invalid-missing-root-rollup)["conformant"] is False` (the binding
     surfaces MS6's `conformant:false` verdict unchanged — a normal return, not an
     exception).
  3. `describe`/`validate` over `invalid/wrong-format-version` **raise** the
     dedicated hard-cut exception (`UnknownFormatVersion`-style), asserted with
     `pytest.raises(...)` — the §0 hard cut preserved end-to-end (never a
     `conformant:false`).
- `crates/python/run_python_tests.sh` (new): the deterministic runner — (1)
  `PYTHON=python3.12 conformance/generator/regenerate.sh` (regenerate-first); (2)
  create/refresh the dedicated `crates/python/.venv` with python3.12; (3)
  `maturin develop` (with the abi3/extension-module wheel) into
  `crates/python/.venv`; (4) install `pytest` into that venv; (5) run `pytest
  crates/python/tests` from that venv. The script states the two venv paths
  explicitly (`conformance/generator/.venv` for generation, `crates/python/.venv`
  for the binding tests) and never mutates the generator venv. `set -euo
  pipefail`; resolves repo root from its own location like `regenerate.sh`.
- `crates/python/README.md` (new): short doc — (a) maturin build/usage
  (`maturin develop` / `maturin build`, `import hdx`, call
  `hdx.describe(path)` / `hdx.validate(path)`); (b) explicit statement that the
  binding **mirrors `hdx-core` and adds no contract logic** (it wraps
  `describe_json`/`validate_json`); (c) the **regenerate-first** prerequisite —
  run `PYTHON=python3.12 conformance/generator/regenerate.sh` before the Python
  tests, because the fixture data is git-ignored; (d) the isolated binding-venv
  path (`crates/python/.venv`) vs the generator venv; (e) the §0 note: the
  wrong-format-version fixture raises the hard-cut exception, never a softened
  `conformant:false`.
- No `hdx-core`/`main.rs` change; no fixture data committed.

**test_plan.**
- Python (`pytest`), run via `crates/python/run_python_tests.sh` after a clean
  regenerate, in `crates/python/.venv`:
  - valid → `describe`/`validate` dicts with schema-matching keys, `conformant`
    True;
