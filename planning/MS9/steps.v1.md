> **Milestone goal (from `planning/milestones.md` MS9).** Last milestone of HDX
> v0.1. Expose `validate` / `describe` to Python via a PyO3 binding in
> `crates/python`, built with maturin — **mirroring, not reimplementing**, the
> `hdx-core` verbs. No contract logic in the binding; it calls `hdx-core` and
> returns the same JSON-shaped results. Closes HDX v0.1.
>
> **Source contract.** `spec/HDX_SPEC.md` §0 (hard version cut), §10 ("a thin,
> JSON-emitting … CLI wraps them … PyO3 mirrors them"), §13 (inert/agnostic),
> §11 (six-field manifest floor). **Planned against** `architecture.md` §2
> (`crates/python` mirrors over the same `hdx-core` API; thin glue, no contract
> logic) and the MS8 fixtures-gitignored / goldens-relocated tracking policy
> (architecture.md Amendments, 2026-06-02).
>
> **Starting state (verified).** Workspace = root bin `hdx` + lib `hdx-core`
> (`members = ["crates/*"]`); `crates/python` does **not** exist yet. `hdx-core`
> already exposes the exact verbs this binding wraps:
> `hdx_core::describe::describe_json(path) -> Result<String, DescribeError>`,
> `hdx_core::validate::validate_json(path) -> Result<String, ValidateError>`, and
> `hdx_core::validate::validate(path) -> Result<ValidationReport, ValidateError>`
> (with `ValidationReport::conformant() -> bool`). Both `DescribeError` and
> `ValidateError` carry a `Manifest(#[source] CoreError)` arm that surfaces the §0

> `schemas/validate.schema.json`. Fixture **data** under `conformance/valid/**`
> and `conformance/invalid/**` is **git-ignored** and regenerated by
> `PYTHON=python3.12 conformance/generator/regenerate.sh`; goldens live in
> `conformance/goldens/`. `python3.12` (3.12.8) and `maturin` are present on the
> build host.

## Scope guard (read first)

MS9 is the **LAST** v0.1 milestone and is deliberately minimal:

- **Mirror, never reimplement (spec §10; architecture §2).** `crates/python` is a
  thin PyO3 wrapper that **calls** `hdx_core::describe::describe_json` and
  `hdx_core::validate::validate_json` (and `validate` for the `conformant` bool)
  and hands the **same** JSON-shaped result back to Python. **No §14 rule logic,
  no contract logic, no re-derived wire shape, no manifest parsing** lives in the
  binding. The Python output is the *exact* JSON the Rust verbs already produce,
  so it matches `schemas/describe.schema.json` / `schemas/validate.schema.json`
  by construction.
- **Expose ONLY `validate` + `describe`.** No `regrid` / `clip` / `reduce`, no
  new `hdx-core` domain or manifest field, no inert-violating field anywhere
  (the binding adds zero new types to the contract). The binding is read-only.
- **§0 hard version cut preserved end-to-end.** The wrong-`format_version`
  fixture MUST raise a clear **hard-cut exception** (surfacing
  `UnknownFormatVersion`) — **never** softened into a `conformant: false` result.
  This is enforced by mapping `*Error::Manifest(CoreError::UnknownFormatVersion …)`
  to a distinct Python exception.
- **No fixture data committed.** The Python tests that read a real dataset MUST
  **regenerate first** (`PYTHON=python3.12 conformance/generator/regenerate.sh`)
  and never commit anything under `conformance/valid/**` or
  `conformance/invalid/**`. All tracked goldens stay under `conformance/goldens/`
  and remain intact after a clean regenerate.
- **Workspace green for the whole workspace.** After every step, `cargo build`,
  `cargo test`, and `cargo clippy --all-targets -- -D warnings` pass for the
  **entire workspace including `crates/python`**, and `maturin build` /
  `maturin develop` into the python3.12 venv succeeds.
- **No `hdx-core` change.** MS9 adds a *new* crate and touches only workspace
  glue + docs (`Cargo.toml`, `architecture.md` amendment); it does **not** modify
  `hdx-core` source, the verbs, the schemas, the generator, or the goldens.

No step below does any later milestone's work (there is none — MS9 is last) and
none introduces a reduction/operation or an inert-violating field.

## Ordering rationale

The dependency chain is: **the crate must compile & link as a cdylib before it
can expose functions; functions must exist before errors can be mapped onto
them; the wrapper must be importable before Python can test it; the §0 hard-cut
behaviour must be wired before a test can assert it.** That yields four
dependency-sequential steps, each independently committable and green:

1. **S1 — Stand up the `crates/python` PyO3 cdylib skeleton + workspace wiring +
   maturin packaging.** Nothing can be wrapped until the crate exists, builds as
   an abi3 `extension-module` cdylib, is a workspace member, and `maturin build`
   succeeds. S1 ships a single trivial exported function (e.g. a version probe)
   purely to prove the import/link/build path end-to-end and to keep the crate
   non-empty and green. No verbs yet → minimal review surface, isolates all the
   toolchain/linking risk into one reviewable commit. This is also where the §2
   build-setup note (abi3 + python3.12) is recorded as an `architecture.md`
   amendment, because that is the load-bearing non-obvious setup.

2. **S2 — Wrap the two verbs (`describe` + `validate`) with faithful error
   mapping, returning parsed `dict`s.** With the crate building, add the two
   thin wrappers that call `describe_json` / `validate_json`, parse the returned
   JSON string into a Python object (`dict`), and map every `hdx-core` error
   onto a Python exception — **with the §0 hard cut (`UnknownFormatVersion`)
   mapped to a distinct, clearly-named exception**, never softened. This is the
   meat of the milestone: it depends on S1 (the crate) and makes the §0
   end-to-end guarantee *implementable* before S3 *asserts* it. Rust-side unit
   coverage (mapping logic) lands here; full Python E2E is S3.

3. **S3 — Python test suite over the regenerated fixtures (regenerate-first).**
   With the wrappers in place, add the pytest/smoke suite that regenerates the
   fixtures first, then asserts: `describe` / `validate` over `valid/minimal`
   return the expected `dict` structures; `validate` over an invalid fixture
   returns `conformant: false`; and `describe` / `validate` over
   `wrong-format-version` **raise** the hard-cut exception. Depends on S2 (the
   functions + exceptions exist). This is the milestone's behavioural proof and
   the regenerate-first policy enforcement.

4. **S4 — `crates/python/README.md` + finalisation.** Documents the maturin build
   + usage, states the binding **mirrors** `hdx-core` (adds no contract logic),
   and records the **regenerate-first** test prerequisite. Last because it
   describes the finished, green crate. Verifies the clean-regenerate invariant
   (goldens intact, no fixture data tracked) as an explicit acceptance gate that
   closes v0.1.

Why not fewer steps? S1 isolates the highest-risk part (PyO3/maturin linking)
into its own green commit so a toolchain surprise never contaminates the verb
wrapping. Why not more? The verbs are two near-identical thin wrappers; splitting
`describe` from `validate` would produce two trivially-similar commits with no
independent reviewability gain. S3/S4 separate *behaviour* (tests) from
*documentation* (README + final invariant check), each a meaningful unit.

## Step MS9-S2 — Wrap `describe` + `validate` with faithful error mapping

**Intent.** Add the two thin Python-callable wrappers — `hdx.describe(path)` and
`hdx.validate(path)` — that **call** `hdx_core::describe::describe_json` /
`hdx_core::validate::validate_json`, parse the returned JSON string into a Python
object (`dict`), and map every `hdx-core` error onto a Python exception. The §0
**hard version cut** (`*Error::Manifest(CoreError::UnknownFormatVersion { .. })`)
is mapped to a **distinct, clearly-named** Python exception — never softened into
a `conformant: false` result. This is the milestone's core: it mirrors the verbs
(no rule logic, no re-derived wire shape) and makes the §0 end-to-end guarantee
implementable. Independently committable: the crate now exposes the two real
functions and stays green; behavioural Python tests follow in S3.

**Changes.**
- `crates/python/src/lib.rs` — add `#[pyfunction] fn describe(path: &str) ->
  PyResult<PyObject>` and `#[pyfunction] fn validate(path: &str) ->
  PyResult<PyObject>`, registered in the `#[pymodule]`. Each:
  - calls the corresponding `hdx-core` verb (`describe_json` / `validate_json`)
    — the binding **never** re-parses the manifest, re-walks the tree, or
    re-derives the wire shape;
  - converts the returned JSON **string** into a Python `dict` (parse the
    canonical Rust wire string with the Python stdlib `json` module via
    `py.import("json")?.call_method1("loads", (s,))`, **or** an equivalent
    `serde_json::Value` → `pyo3` conversion **kept strictly as a transport**, not
    a reshape) so the returned `dict` is byte-for-byte the MS5/MS6 JSON;
  - maps errors via a single private helper (see below).
- `crates/python/src/lib.rs` (errors) — define Python exception types via
  `pyo3::create_exception!`, e.g. a base `HdxError`, a `UnknownFormatVersionError`
  (the §0 hard cut), and an `HdxIoError` / `HdxValidationError` family as needed,
  plus a private `fn map_describe_err(e: DescribeError) -> PyErr` and
  `fn map_validate_err(e: ValidateError) -> PyErr` that:
  - match `Manifest(CoreError::UnknownFormatVersion { found })` → raise
    `UnknownFormatVersionError` carrying the offending version string (the §0
    hard cut, distinct exception, **never** softened);
  - map `ManifestUnreadable` / other `Manifest(_)` (malformed manifest) and
    `Discovery(_)` / `Serialize { .. }` to the appropriate `HdxError`-family
    exception with the error's `Display` message.
  - No silent fallback: every error arm raises.
- `crates/python/src/lib.rs` (module doc `//!`) — one-line purpose + the explicit
  statement that the binding **mirrors** `hdx-core` and adds no contract logic.
- No `hdx-core` change. No new manifest/domain field. The binding still exposes
  **only** `describe` + `validate` (plus the S1 trivial probe, optional).

**Test plan.**
- Rust unit tests in `crates/python` for the **error-mapping helpers** in
  isolation (constructing a `DescribeError::Manifest(CoreError::UnknownFormatVersion
  { found: "0.2".into() })` and asserting `map_describe_err` produces the
  `UnknownFormatVersionError`-typed `PyErr`; same for `ValidateError`). These run
  under `cargo test` and need a `Python::with_gil` context, gated to the test
  build. (Full Python-level E2E is S3.)
- `cargo clippy --all-targets -- -D warnings` clean (no `unwrap`/`expect` in the
  mapping paths beyond test code; the wrappers return `PyResult`).
- `maturin build` still succeeds with the two new exported functions.

**Acceptance.**
- `cargo build`, `cargo test`, `cargo clippy --all-targets -- -D warnings` pass
  for the whole workspace.
- `hdx.describe` / `hdx.validate` are exported and each is a **thin wrapper** that
  only calls `describe_json` / `validate_json`, transports the JSON to a `dict`,
  and maps errors — verified by review (no §14 rule, no manifest parse, no wire
  reshape in `crates/python`).
- The §0 hard cut is mapped to a **distinct** Python exception
  (`UnknownFormatVersion`-style), never softened to a report — verified by the
  Rust mapping unit test.
- `maturin build` succeeds with the verbs exported.
- No `hdx-core` change; no `regrid/clip/reduce`; no inert-violating field; only
  `validate` + `describe` exposed.
- Spec MUST-checks: **M1/M2** (the §0 hard version cut) preserved through the
  binding's error mapping — the wrong-version path raises, never reports
  conformant. No new check IDs (binding is a mirror). §13 inertness preserved.
- Commit via the bump+tag convention.

**Spec refs.** §0 (hard version cut preserved end-to-end), §10 (PyO3 mirrors the
verbs; same JSON results), §11 (binding declares no manifest field), §13
(inert/agnostic); architecture §2 (mirror over the same API, no contract logic);
R4 (same stable JSON wire shape, reused not re-derived).

**Commit message.** `feat(python): expose validate/describe with hard-cut error mapping`

## Step MS9-S3 — Python test suite over regenerated fixtures (regenerate-first)

**Intent.** Add the Python behavioural test suite (pytest, or a self-contained
smoke script) that proves the binding mirrors the verbs end-to-end over **real**
data, enforcing the **regenerate-first** policy. Depends on S2 (the functions +
exceptions exist). This is the milestone's behavioural proof: `describe` /
`validate` over the regenerated valid fixture return the expected `dict`
structures; `validate` over a regenerated invalid returns `conformant: false`;
`describe` / `validate` over the wrong-`format_version` fixture **raise** the
hard-cut exception (mirroring §0). Independently committable: it adds tests +
their runner glue only; no source change to the binding or `hdx-core`.

**Changes.**
- `crates/python/tests/` (e.g. `test_mirror.py`) — pytest tests that:
  - import the `hdx` module (requires `maturin develop` into the python3.12 venv —
    the runner script does this);
  - `hdx.describe(<valid/minimal>)` returns a `dict` whose top-level keys are
    `{manifest, basins, fields, grids, time_extents, delineations}` (the MS5 wire
    shape) and whose `manifest.format_version == "0.1"` — asserting structure, not
    re-deriving it; optionally assert the `dict` round-trips equal to the
    committed `conformance/goldens/valid-minimal.describe.json` to prove the
    Python output matches the Rust wire shape exactly;
  - `hdx.validate(<valid/minimal>)` returns a `dict` with `conformant == true` and
    a `checks` array (the MS6 wire shape);
  - `hdx.validate(<invalid/missing-root-rollup>)` (a regenerated invalid) returns
    a `dict` with `conformant == false`;
  - `hdx.describe(<invalid/wrong-format-version>)` **raises** the hard-cut
    exception (`UnknownFormatVersion`-style), and likewise `hdx.validate(<…/wrong-
    format-version>)` raises — **never** returns `conformant: false` (the §0

  This script is the documented, reproducible Python-test entry point.
- Fixture paths in the tests resolve relative to the repo root
  (`conformance/valid/minimal`, `conformance/invalid/missing-root-rollup`,
  `conformance/invalid/wrong-format-version`) — all under the **git-ignored**

  git.**
- No change to `conformance/generator/`, the goldens, the schemas, or `hdx-core`.

**Test plan.**

  `pytest`): all Python tests pass — describe/validate dicts match the expected

**Acceptance.**
