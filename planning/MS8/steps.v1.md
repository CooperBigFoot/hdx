## 0. Grounding — the enforced-vs-skipped reconciliation (READ FIRST)

This milestone is grounded in the **actual** §14 behavior of `crates/core/src/validate.rs`
as it ships after MS6 — not an assumed behavior. The fold-in CRITICAL requirement is that
MS8 first determines, *from the code*, the exact set of §14 checks that **RUN and can yield
`conformant:false`** (an enforced `ran:fail`) versus those reported **R3-SKIPPED** in v0.1
(always `status:"skipped"`, never `ran:fail`, never flipping the verdict). The set below was
read directly from `validate.rs` (`build_report`, the per-check `check_*` functions, and the
test module's MS8-deferral comment).

### 0.1 The three outcome buckets (the spine of every MS8 fixture)

A one-mutation invalid fixture, fed through `validate(path)`, lands in **exactly one** of
three buckets. The bucket dictates what the regression test asserts. This classification is
load-bearing because several checks **fail closed during discovery** (the reader returns a
typed `Err` *before* the report is built), so their negative is a `ValidateError`, not a
`conformant:false` report.

**Bucket A — verb returns `Err(ValidateError::…)` (fail-closed at entry or discovery).**
The mutation trips the §0 entry gate or a metadata reader that fails closed, so `validate`
returns `Err` and **never produces a report at all**. The regression test asserts the exact
`ValidateError` variant (and the wrapped `CoreError`). These are still genuine §14 negatives —
the dataset is non-conformant — they are just surfaced as a typed error rather than a
`ran:fail` outcome (the load-bearing report-vs-error split documented in `validate.rs`).

| Check | Mutation | Why it is `Err`, not `conformant:false` |
|---|---|---|
| **M2** | `format_version: "0.2"` | §0 hard cut in `Manifest::from_json` → `Err(Manifest(UnknownFormatVersion))` **before** discovery (reuses MS2's `wrong-format-version`). |
| **M3** | a 7th manifest field | `deny_unknown_fields` boundary parse → `Err(Manifest(ExtraManifestField))`. |
| **M4** | empty `crs` / bad `created_at` / empty `cadence` | manifest boundary parse → `Err(Manifest(EmptyCrs / InvalidTimestamp / EmptyCadence))`. |
| **I1** (outlines leg) | drop `basin_id` from `outlines.geoparquet` | geoparquet reader requires the three columns → `Err(Discovery(MissingGeometryColumn{column:"basin_id"}))`. |
| **G3** | strip CF `grid_mapping`/CRS from a Zarr (or georef tags from a COG) | the gridded reader requires resolvable georef → `Err(Discovery(MissingGridGeoref))`. |
| **Geo1** (missing-column leg) | drop `delineation` (or `geometry`) from `outlines` | geoparquet reader requires the three columns → `Err(Discovery(MissingGeometryColumn))`. |

**Bucket B — verb returns a report with `conformant:false` and EXACTLY one `ran:fail`.**
The mutation survives discovery (every reader succeeds) but the rule function over the typed
model records a `Fail`. The regression test asserts `report.conformant() == false`, the pinned
check is `status:Ran, result:Fail` (with a detail substring), and **every other check is
`pass` or an honest `skipped`** (the one-violation-purity assertion).

| Check | Mutation | Rule that records the fail |
|---|---|---|
| **L1** | delete one root rollup (`outlines.geoparquet`) | `check_l1` (reuses MS2's `missing-root-rollup`). |
| **L2** | delete a basin's `scalar_dynamic.parquet` (forward leg) | `check_l2` (a basin with no `time` descriptor). |
| **I2** | rewrite one basin's in-file `basin_id` to disagree with its `basin=<id>` folder | `check_i2`. |
| **I3** | give two basins the **same** in-file `basin_id` | `check_i3`. |
| **H1** | change one basin's field schema (e.g. a dtype, or a renamed/extra field) | `check_h1`. |
| **H2** | give one basin a divergent grid-label set (e.g. rename its `era5` artifacts to `chirps`) | `check_h2`. |
| **T1** | make one `scalar_dynamic` `time` non-monotonic (or rename / make nullable / mis-type) | `check_t1`. |
| **G2** | keep the shared `era5` label across COG+Zarr but shift one artifact's extent/resolution so they are **not** cell-for-cell aligned | `check_g2`. |
| **M5** | set one georeferenced file's CRS to a different EPSG than the manifest `crs` | `check_m5`. |

**Bucket C — R3-SKIPPED checks: NO enforceable v0.1 negative exists.** For these, MS8 does
**NOT** claim a `conformant:false` negative. Instead it **documents** the absence and
(optionally) ships a *targeted* fixture asserted to be **reported skipped-with-reason and
still conformant**. These checks are `status:"skipped"` on EVERY dataset (including the
valid baseline) because their decisive leg is byte-deep / on-disk-shape-dependent and v0.1
discovery does not surface it.

| Check | Why R3-skipped in v0.1 (verbatim from `validate.rs`) | MS8 treatment |
|---|---|---|
| **L3** | `check_l3` always `Skipped` (`ByteDeep`): the absence-is-NaN-not-a-missing-file leg needs a byte-deep cell-payload read, which `validate` never performs (LOW-3). | Document "no enforceable v0.1 L3 negative." Optional fixture: a tree with a NaN-encoded gap (a value-shape, not a file-shape) asserted **still conformant**, L3 still `skipped`. No stray-file fixture that would change the verdict. |
| **M6** | `check_m6` rule (b) (per-basin axis **regularity**) always `Skipped` (`ByteDeep`): two-point `[start,end]` extent cannot prove a constant interior step; rule (a) cadence-non-empty is M4's domain and an empty cadence is rejected at the manifest boundary, so M6 cannot `ran:fail` through the verb. **No cross-basin same-step rule exists (MED-1 resolution).** | Document "no enforceable v0.1 M6 negative." The targeted "irregular-axis" fixture's expected outcome is **M6 reported `skipped`-with-reason, dataset still conformant** — assert exactly that. Do NOT resurrect the dropped cross-basin same-step rule. |
| **T2** | `check_t2` always `Skipped` (`ByteDeep`): cross-artifact full time-axis identity needs both 1-D axes; v0.1 surfaces only the scalar `[start,end]` extent. | Document "no enforceable v0.1 T2 negative." Optional fixture: a tree whose Zarr `time` differs from the scalar `time` asserted **still conformant** (T2 `skipped`), proving the verdict does not depend on the unreadable leg. |

### 0.2 G1 reachability note (recorded, not assumed)

`check_g1` runs and *can* `ran:fail` (a gridded field with no grid label). But the in-memory
domain type makes a label-less gridded `Field` unrepresentable (`Field::new` rejects it), and
the discovery readers only mint gridded fields *with* a label. The on-disk falsifiable form of
G1 is **a positional / unnamed band or CF variable** (no self-naming). MS8 attempts a
"`gridded·dynamic` Zarr variable with no name / a positional channel axis" fixture; **its actual
outcome is determined empirically when the fixture lands** and recorded as an architecture
amendment:
- if the reader admits the unnamed band but discovery surfaces it without a label → `check_g1`
  `ran:fail` (Bucket B);
- if the reader fails closed on the malformed variable → `Err(Discovery(…))` (Bucket A);
- if a self-naming Zarr/COG cannot represent "no name" at all → G1 has **no on-disk negative**,
  documented like Bucket C (G1 stays `ran:pass`, the self-naming-by-construction guarantee is
  the proof). The plan does **not** pre-commit to a bucket for G1; the step records the
  observed outcome and the README table reflects it.

### 0.3 The enforced-vs-skipped split, stated for the README

- **Enforced (can yield a non-conformant verdict — Bucket A `Err` or Bucket B `ran:fail`):**
  M2, M3, M4, M5, L1, L2 (forward leg), I1, I2, I3, H1, H2, T1, G2, G3, Geo1 (and G1 pending
  the §0.2 empirical result). M1 is the "manifest is valid JSON / version read first" gate,
  proven by the M2/M3/M4 negatives + the valid baseline.
- **R3-SKIPPED (no enforceable v0.1 negative):** L3, M6 (axis-regularity leg), T2.

This split is stated explicitly in `conformance/README.md` (MS8-S6) per the fold-in.

## 1. Ordering rationale

The steps are dependency-sequential and each leaves the repo **green** (`cargo build` +
`cargo test` + `cargo clippy --all-targets -- -D warnings`).

1. **S1 — generator mutation framework + README split first.** Before any fixture exists, the
   generator needs a *typed, closed* invalid-catalog (extending `mutate.py`'s `Invalid` enum)
   and the per-mutation surgical-edit + "differs in exactly one way" self-assertion machinery,
   plus the enforced-vs-skipped split written into `conformance/README.md`. This is the scaffold
   every later step plugs into; committing it first means each subsequent fixture step is a
   small, reviewable addition. No Rust change → trivially green.

2. **S2 → S5 — fixtures grouped by *bucket and family*, smallest blast radius first.** The
   fixtures are added in dependency-light groups so each step is independently reviewable and
   each commit regenerates a deterministic, self-asserting tree:
   - **S2 — Manifest/identity Bucket-A + Bucket-B family** (M3, M4, I1-outlines + I2, I3): the
     manifest mutations and the parquet/geoparquet identity mutations, which only touch
     `manifest.json` and parquet columns (the mature, low-risk path).
   - **S3 — Layout + homogeneity family** (L1 [reuse], L2, H1, H2): tree-shape and schema-shape
     mutations.
   - **S4 — Time + grid family** (T1, G2, M5, G3, Geo1-missing-column, and the G1 probe): the
     parquet `time` mutation plus the Zarr/COG georef/alignment mutations (the
     highest-uncertainty path, last among the enforced fixtures).
   - **S5 — R3-skip targeted fixtures + documentation** (L3, M6-irregular-axis, T2): the
     Bucket-C "skipped-with-reason, still conformant" fixtures.
   Each fixture step regenerates the whole suite deterministically and adds its Rust regression
   assertions in the same commit, so the suite is never half-wired.

3. **S6 — the consolidated regression matrix + goldens + README finalization.** With every
   fixture present, S6 adds (a) the table-driven regression test that runs `validate` over
   **every** invalid fixture and asserts its bucket/outcome + one-violation purity in one place;
   (b) the extended golden outputs (the describe golden + the validate golden for the valid
   fixture, plus the documented golden-update workflow); (c) the companion-mask /
   `{source}_{variable}` ordinariness re-assertion across both `describe` and `validate`; and
   (d) the finalized `conformance/README.md` check-id → fixture table and golden-update
   workflow. S6 is last because it depends on all fixtures and consolidates the proof.

**Why fixtures-then-matrix rather than one-fixture-one-commit:** ~16+ fixtures as 16 separate
commits would be noisy and each would touch the same generator dispatch + README table. Grouping
by family (S2–S5) keeps each commit a coherent, reviewable unit (one writer-side mutation family +
its Rust assertions) while the consolidating matrix (S6) lands once, at the end. Every group step
is independently green and committable.

## 2. Scope guard

No step in MS8 implements a later milestone's work or violates the inert/agnostic discipline:

- **No MS9 (PyO3) work** — MS8 touches only `conformance/`, `crates/core` *tests*, the goldens,
  and `conformance/README.md` (plus, at most, a recorded architecture amendment for the G1 probe
  result). No `crates/python`.
- **No new `hdx-core` rule logic** — MS8 adds **no** `check_*` function and **no** change to
  `build_report`. It asserts against the MS6 validator exactly as shipped. (If a fixture reveals a
  genuine MS6 validator bug, that is flagged as a finding, not patched inside MS8 — MS8's contract
  is fixtures + tests + docs.)
- **No regrid/clip/reduce, ever** — the generator emits bytes; it performs no hydrology operation,
  no reduction, no resampling. (Reaffirmed in `conformance/README.md` Rule 1, already present.)
- **No manifest mutation beyond the floor** — the M3 fixture *adds a 7th field to demonstrate
  rejection*; it does not propose a new manifest field. The manifest stays exactly the six floor
  fields everywhere a conformant tree is expected.
- **No new domain field anywhere** — the companion-mask (`{field}_was_filled`) and
  `{source}_{variable}` fields remain **ordinary** catalog entries; MS8 re-asserts they carry no
  `mask`/`companion`/`source`/`variable`/`belongs_to`/`role` key in both `describe` and
  `validate`. No type or field gains a transform/role/semantic/provenance attribute.
- **Every invalid is one surgical mutation off the single valid baseline** (LOW-2), generated
  **programmatically**, never hand-edited; the "differs in exactly one way" self-assertion guards
  it at generation time, and the "exactly this id fails, others pass/skip" Rust assertion guards
  against cross-contamination.
