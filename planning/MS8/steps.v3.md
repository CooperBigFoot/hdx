# MS8 — Conformance suite: exhaustive invalids + golden outputs (resolves R2, part 2)

> **Milestone goal (from milestones.md MS8).** Complete **R2**: build the curated
> conformance suite — the **exhaustive family of invalid fixtures, each pinning exactly
> one violated §14 check id** (the negative matrix MS6 deferred), plus golden outputs —
> all regenerated by the MS2 generator and wired into regression tests. This is the
> validator's full fail-closed proof across the §14 checklist.
>
> **Scope is fixtures + tests + docs ONLY.** MS8 adds **no** Rust contract logic, **no**
> manifest mutation, **no** new domain field, and **never** introduces regrid/clip/reduce.
> Every invalid fixture is generated **programmatically from the single valid baseline via
> exactly one surgical mutation** (LOW-2). The validator (`hdx-core`) is frozen as of MS6;
> MS8 only feeds it new on-disk trees and asserts the verdicts.
>
> **"Extend the golden" is satisfied by extending the golden SUITE, not by rewriting the
> valid-fixture golden FILES.** The determinism contract (MS2) + the committed snapshot
> tests (`validate_json_equals_committed_golden`, `golden_validates_against_validate_schema`,
> `describe_json_equals_committed_golden`, `golden_validates_against_describe_schema`)
> already lock the valid-fixture goldens **byte-for-byte**; MS8 adds no field and rewrites
> no baseline bytes, so those two golden FILES stay byte-unchanged. The deliverable
> "Golden describe outputs … (extending MS5's golden)" is met by (a) adding the MS8
> negative/skip golden matrix (one committed `validate` report per Bucket-B invalid) around
> the existing byte-unchanged goldens, (b) re-affirming the valid-fixture goldens with
> snapshot regression, and (c) documenting the golden-update workflow for the new family —
> i.e. extending the golden suite, not editing the valid baseline. (This reconciles the
> literal milestone wording with the determinism contract; see the MEDIUM fold below.)

## Grounding: the enforced-vs-skipped reconciliation (read `validate.rs` FIRST)

This milestone is grounded in the **actual** `crates/core/src/validate.rs` `build_report`
(validate.rs:1327-1396) and the readers it consumes (`discovery.rs`, `gridded_discovery.rs`,
`scalar_reader.rs`, `geoparquet_reader.rs`). The §14 checks fall into **three buckets**;
MS8 builds a one-violation invalid fixture **only for the enforced Bucket-B set**, and
documents the others honestly.

### Bucket A — entry-gate `Err` (a `ValidateError`, NOT a `conformant:false` report)

`M1, M2, M3, M4` are folded into `Manifest::from_json` and evaluated **before** `discover`
(validate.rs:1272-1276). A violation returns `Err(ValidateError::Manifest(...))` /
`Err(ValidateError::ManifestUnreadable)`, **never** a `conformant:false` report (the §0 hard
cut + boundary parse precede discovery by construction). Their on-disk negatives are
asserted as **`Err` outcomes**, not as `conformant:false` reports:

| Check | One-mutation invalid | Verb outcome on the tree |
|---|---|---|
| **M2** | `invalid/wrong-format-version/` (MS2, reused) | `Err(Manifest(UnknownFormatVersion{found:"0.2"}))` |
| **M3** | `invalid/extra-manifest-field/` (new) | `Err(Manifest(ExtraManifestField{field}))` |
| **M4** | `invalid/empty-cadence/` (new) | `Err(Manifest(EmptyCadence))` |

> M1 (manifest unreadable / not valid JSON) is the `ValidateError::ManifestUnreadable` /
> JSON-parse path; it is already exercised in-tree by MS6's
> `entry_gate_reports_unreadable_manifest_for_missing_manifest_json`. MS8 adds **no** new M1
> tree (an absent/garbage `manifest.json` is not "one surgical mutation off the baseline
> that pins one §14 check" — it is the structural entry failure MS6 already covers). M1 is
> documented as a Bucket-A entry-gate case with no MS8 tree.

> **The HIGH fix (I3), stated explicitly.** A prior iteration listed `I3_DUPLICATE_BASIN_ID`
> as a Bucket-B fixture. That is **wrong against the real validate.rs**: `check_i3`'s only
> input is the per-basin `scalar_dynamic` `.first()` in-file id (`in_file_basin_ids`,
> validate.rs:1439-1446, sourced from discovery.rs:261). Mutating `scalar_static` rows has
> **zero** effect on I3, and the only way to put a duplicate into `in_file_basin_ids` is to
> set two basins' `scalar_dynamic` in-file id to the same value — which necessarily trips
> `check_i2` first (it runs unconditionally and at most one of the two basins can match its
> `basin=<id>` folder). So I3 has **no clean one-violation v0.1 negative**; intra-file
> duplicates are invisible to discovery. **I3 is reclassified as a documented
> no-enforceable-v0.1-negative finding** (mirroring G1/G3). An enforceable I3 negative would
> require a discovery/`build_report` change (surfacing `scalar_static` row `basin_id`
> values) which is **OUT of MS8 scope** — raised as a finding, never patched here. I3's
> positive path is covered by the valid baseline (`check_i3` `ran:pass`).
>
> The S6 consolidated matrix and every coverage table below therefore contain **no
> Bucket-B I3 row**; I3 appears only as a no-negative finding row.

### The collision-purity proofs (why each Bucket-B mutation stays one-violation)

The risk is a mutation accidentally tripping a *second* check. Verified against the rules:

- **I2 (folder mismatch) stays I2-only.** `fields_by_basin` (validate.rs:1406) compares
  schema, not id values, so H1 is unaffected; `in_file_basin_ids` feeds I3 but a single
  mismatched-but-distinct id is still **distinct** (no duplicate ⇒ I3 pass). Only `check_i2`
  fails. Pinned mutation: basin 0002's `scalar_dynamic` `basin_id` value `0002` → `9002`
  (distinct, wrong folder).
- **H1 (ragged schema) stays H1-only.** Per-basin schema is the `scalar_dynamic` field list
  (`fields_by_basin`, validate.rs:1406, sourced from `BasinScalar::fields`,
  discovery.rs:156). Renaming the dynamic data column in **one** basin's
  `scalar_dynamic.parquet` (`streamflow` → `streamflow_alt`) diverges that basin's schema ⇒
  `check_h1` `ran:fail`. It does **not** touch the grid-label set (H2), `basin_id`/`time`
  (I/T), or the gridded catalog (G1). One violation.
- **H2 (divergent grid label) stays H2-only.** `labels_by_basin` (validate.rs:1422) is the
  per-basin static⊕dynamic grid-label **set**. Renaming **both** of one basin's gridded
  artifacts `era5`→`era5b` (so that basin's COG/Zarr still share a label — G2 still passes
  intra-basin) makes that basin's label set `{era5b}` ≠ the reference `{era5}` ⇒ `check_h2`
  `ran:fail`. G2 is per-basin (validate.rs:1113) so the renamed basin still aligns; H1
  (scalar schema) is unchanged. One violation. **Pinned mutation: rename BOTH
  `basin=0002/gridded_static/era5.tif`→`era5b.tif` AND
  `basin=0002/gridded_dynamic/era5.zarr`→`era5b.zarr`, leaving its geometry untouched so its
  own COG/Zarr still coincide.**
- **T1 (unsorted time) stays T1-only.** Writing one basin's `scalar_dynamic` `time` column
  **descending** (and clearing the `SortingColumn` footer metadata) makes
  `time.is_sorted_ascending()` false ⇒ `check_t1` `ran:fail` (validate.rs:667). The
  descriptor stays named `time`, `Timestamp`, non-nullable; the data column, `basin_id`,
  schema, and grids are untouched. The Zarr `time` axis would diverge in order — but **T2 is
  R3-skipped** (check_t2 always `Skipped`, validate.rs:1090), so it cannot co-fail. One
  ran-fail. The named test asserts "discovers OK, then T1 ran:fail" (mirroring MS6's
  `t1_negative_per_leg`).
- **L2 (missing required gridded artifact) stays L2-only — PINNED to the gridded-subtree
  form.** The provably-L2-only mutation is to delete **one** basin's `gridded_dynamic/`
  subtree: it leaves the scalar schema intact (H1/T1/I pass), the basin still has its
  `scalar_dynamic` (so the scalar-dynamic L2 leg passes), and `check_l2`'s gridded-dynamic
  leg fires (`dynamic_artifacts().is_empty()` while the field set declares `GriddedDynamic`,
  validate.rs:821) ⇒ `ran:fail`. **Collision note (the rejected alternative):** deleting a
  basin's `scalar_dynamic.parquet` *also* makes that basin's `fields_by_basin` entry `(id,
  [])`, which differs from the reference schema ⇒ H1 would **co-fail**. That form is
  therefore rejected; S3 records it as the rejected alternative in a code comment + README.
  The gridded-subtree-deletion form is pinned because it is provably L2-only. (H2 is
  unaffected: deleting one basin's whole dynamic subtree empties its dynamic-label set —
  which could trip H2; **so the pinned mutation deletes only the single `<label>.zarr` store
  inside `gridded_dynamic/` but keeps the `gridded_dynamic/` directory empty is NOT valid**
  — instead the mutation deletes the `gridded_dynamic/` subtree of a basin AND the test
  asserts the outcome empirically: if H2 co-fails because that basin's label set becomes
  empty, S3 falls back to the **`gridded_static/` (COG) subtree deletion** which keeps the
  dynamic Zarr label `{era5}` intact for H2 while still firing `check_l2`'s gridded-static
  leg, validate.rs:811.) **S3 pins the COG-subtree-deletion form as primary** (it keeps the
  Zarr `era5` label so H2's per-basin set stays `{era5}`… — see the H2 collision caveat
  below) **and verifies one-violation purity empirically before committing**, downgrading to
  whichever of {delete `gridded_static/`, delete `gridded_dynamic/`} the regression proves
  L2-only on the actual `check_l2`/`check_h2`/`check_h1` rules.
  - **H2-collision caveat for L2:** `labels_by_basin` unions a basin's static + dynamic
    labels (validate.rs:1428). Deleting `gridded_static/` leaves the basin's label set =
    `{era5}` (from the Zarr) — **identical** to the reference ⇒ H2 passes; deleting
    `gridded_dynamic/` leaves `{era5}` (from the COG) — also identical ⇒ H2 passes. So
    **either** deletion keeps the basin's label set `{era5}` (because both subtrees use the
    same shared label) and H2 is unaffected. The **only** L2-only-vs-H1 distinction is the
    scalar deletion (rejected). The pinned form is therefore **delete `gridded_dynamic/`** of
    a basin; the test asserts L2 `ran:fail` with H1/H2/T1/I all `ran:pass`.
- **M5 (crs mismatch) stays M5-only.** Rewriting the manifest `crs` to `EPSG:3857` while the
  files keep `EPSG:4326` makes `check_m5` find a grid CRS ≠ manifest ⇒ `ran:fail`
  (validate.rs:978). The manifest still parses (non-empty crs ⇒ M4/M6 pass), the files are
  untouched (G3/G2/H2 pass). The mutation is the manifest-only crs string. One violation. The
  test asserts "discovers OK, then M5 ran:fail".
- **G2 (misaligned shared grid) stays G2-only.** Shifting **one** basin's Zarr `lon`/`lat`
  origin by one cell (so the Zarr extent differs while width/height/res are unchanged and the
  store still decodes) makes the COG and Zarr `era5` extents disagree ⇒ `check_g2` `ran:fail`
  (validate.rs:1127). The label is still shared (H2 unaffected: the label set is still
  `{era5}`), the store still georeferences (G3 pass), and the Zarr still decodes (no
  discovery `Err`). One violation. **Pinned mutation: shift basin 0001's Zarr
  `GRID_WEST`/`GRID_NORTH` by +`GRID_RES` so `lon[0]`/`lat[0]` move one cell; the COG keeps
  the baseline origin; width=6/height=8/res=0.25 unchanged.** The test asserts "discovers OK,
  then G2 ran:fail" (mirroring `t1_unsorted_time_pins_exactly_t1`) and that the COG/Zarr
  `era5` extents now differ (the negative of MS6's `g2_positive_path_fires_*`).
- **Geo1 (outlines partitioned) stays Geo1-only — CONDITIONAL.** The realizable Geo1
  `conformant:false` form is a `outlines.geoparquet/` directory partitioned
  `delineation=<label>/…` recorded with `partitioned_by_delineation()==true` ⇒ `check_geo1`
  `ran:fail` (validate.rs:1210). **But `geoparquet_reader.rs` currently hard-codes
  `partitioned_by_delineation==false` for the single root file it is handed
  (geoparquet_reader.rs:147).** S4 **first verifies empirically** whether `discover` over a
  partitioned-outlines tree surfaces an `OutlinesInfo` with `partitioned_by_delineation()==
  true`. If it does, the fixture + `geo1_partitioned_pins_exactly_geo1` test land. **If the
  reader cannot represent a partitioned outlines as `true` without a Rust change, Geo1's
  negative is OUT of v0.1 scope** (a Rust change is not MS8 work): Geo1 is downgraded to a
  documented finding, its positive path (baseline) + the absent-outlines honest-skip (MS6's
  `missing_root_rollup` test) remain its only verb-level behaviours.

## Ordering rationale

MS8 is fixtures-then-wiring-then-docs, layered so the repo stays green after every step and
each step is one reviewable conventional commit:

1. **S1 — generator scaffolding + valid-baseline golden re-affirm (no new fixtures yet).**
   Extend the generator's mutation framework (a closed `Invalid`-enum-style registry of MS8
   mutations) and re-affirm the byte-unchanged valid-fixture goldens (describe + validate)
   with snapshot tests, so the golden-suite extension has a stable base. This isolates the
   framework change from fixture content and proves the determinism contract holds before
   mutations multiply.

2. **S2 — Bucket-A `Err`-asserting manifest invalids (M3, M4) + reuse M2.** The simplest
   mutations (manifest JSON edits) with the simplest assertions (`Err` form, no report);
   reader-independent.

3. **S3 — Bucket-B scalar-side invalids (L2, I2, T1, H1).** Mutations confined to the
   scalar/parquet half + layout; each yields one `conformant:false` `ran:fail` + a committed
   golden report.

4. **S4 — Bucket-B grid/geometry invalids (H2, M5, G2; Geo1 conditional).** Mutations on the
   gridded/geometry half; S4 also empirically resolves the Geo1 partitioned-outlines reader
   question and downgrades Geo1 to a finding if the reader hard-codes `false`.

5. **S5 — Bucket-C reported-skip fixture (M6) + the no-negative findings.** The
   `irregular-time-axis` fixture asserted **reported-skipped + still conformant** (M6), plus
   the documented findings for I3/G1/G3/I1/L3/T2/M1 (no `conformant:false` claim).

6. **S6 — the consolidated regression matrix + `conformance/README.md` rewrite + golden
   workflow.** One exhaustive parametrized regression iterating every committed fixture, the
   full check-id → outcome table (three-bucket split explicit, **no Bucket-B I3 row**), the
   golden-update workflow, and the companion-mask/`{source}_{variable}` ordinariness
   re-assertion. Last because it depends on every fixture existing.

Dependencies: **S1 → S2 → S3 → S4 → S5 → S6.** S2–S5 each commit independently and leave the
repo green; S6 consolidates. Every step ends with `cargo build` + `cargo test` +
`cargo clippy --all-targets -- -D warnings` green, the generator regenerating deterministically
(`regenerate.sh` self-assertions pass), and the bump+tag commit convention.

## Scope guard

- **No step exceeds MS8 or does a later milestone's work.** MS9 (PyO3) is untouched; MS8
  adds only fixtures (`conformance/`), regression tests (`crates/core/src/*.rs` test modules
  + the generator), and docs (`conformance/README.md`). No CLI change (MS7 owns the CLI; MS8
  may *invoke* the existing verbs in tests but adds no subcommand).
- **No inert/agnostic violation.** No fixture adds a manifest field beyond the six floor
  fields; the M3 invalid *deliberately* adds a seventh field **to be rejected**, never to be
  carried. No type/field anywhere gains transform/role/semantic/provenance. The
  companion-mask (`era5_precipitation_was_filled`) and `{source}_{variable}`
  (`era5_precipitation`) fields stay **ordinary** in both `describe` and `validate`,
  re-asserted in S6 (no special handling).
- **No regrid/clip/reduce, no new domain field, no manifest mutation of the floor, no
  reader/`build_report` change.** The validator is frozen at MS6. If a desired negative
  cannot be realized without a Rust change (I3, G1, G3, I1, possibly Geo1, L3, T2, M6
  regularity), it is raised as a **finding**, never patched — the §14 enforcement-depth note
  (spec §14) sanctions honest skips + the R3 classification.
- **Every invalid is one surgical mutation off the single valid baseline (LOW-2)**, generated
  programmatically and confirmed by the "differs in exactly one way" self-assertion; no tree
  is hand-edited.
- **The valid-fixture golden FILES stay byte-unchanged** (determinism contract); MS8 extends
  the golden SUITE (adds negative-report goldens + re-affirms the baseline goldens), never
  rewrites the baseline goldens.
