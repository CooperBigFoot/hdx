## Issues filed (all low severity — polish, not blockers)

### 1. (low, convention) H2 mutation is one *semantic* mutation but many *files*

S4's `DIVERGENT_GRID_LABEL_SET` (H2) adds a whole `chirps.zarr` + `chirps.tif` pair to one
basin. A Zarr store is a directory of many files, so the existing
`assert_differs_in_exactly_one_way` invariant (`_require(not added, ...)`,
assertions.py:777) — which today *forbids any added file* — must be widened to accept this
variant's multi-file added set. The plan says "extend `assert_differs_in_exactly_one_way`
for the three new variants" (S4), so it is in scope, but the prose frames every mutation as
"one surgical mutation" / "rewrites **one** artifact"; the H2 case is one *added grid label*
realized as several new files, not a one-file rewrite. **Suggested fix:** in S4, state
explicitly that the H2 variant's self-assertion asserts the added-file set is exactly the
new label's artifact pair (and the removed/changed sets are empty), so "one surgical
mutation" reads as "one semantic grid-label addition" — keeping the LOW-2 invariant honest
and non-vacuous.

### 2. (low, convention) M3 generator mutation must not reuse the `MANIFEST_FIELDS`-only rewrite

S2's `_mutate_extra_field` (M3) must *add* a 7th key to `manifest.json`. The existing
`_mutate_format_version` rewrites the manifest as `{field: obj[field] for field in
MANIFEST_FIELDS}` (mutate.py:97), which would silently *drop* any extra key. If the new
mutation copies that idiom it produces a 6-field manifest and fails to pin M3.
**Suggested fix:** note in S2 that `_mutate_extra_field` writes the six floor fields **plus**
the one extra key (not the floor-only reorder), and that the M3 self-assertion asserts the
key set is the six floor fields plus exactly the one extra — the plan describes this for the
assertion side but not the writer side.

### 3. (low, vague-acceptance) S5 describe-golden helper parameterization is implied, not stated

The existing describe.rs `golden_value()` (describe.rs:1126) is hardcoded to
`valid/minimal/describe.golden.json`. S5 adds describe goldens for two *other* fixtures and
says "with a per-fixture golden helper" but does not state that the existing single-fixture
helper is generalized (or a parameterized helper added) without disturbing the four baseline
snapshot/schema tests S1 re-affirms. **Suggested fix:** in S5, state that the new describe
snapshot tests use a `describe_golden(fixture_rel)` helper (or a parameterized form) that
leaves the existing `describe_json_equals_committed_golden` /
`golden_validates_against_describe_schema` baseline tests byte-unchanged — so S1's green
floor is provably preserved.

### 4. (low, vague-acceptance) "empirically confirm before committing each golden" is a manual gate, not an automated assertion

S3/S4 say to "empirically confirm before committing each golden that I1's drop-form ... is
I1-only" and similarly for I3/M5/G2/H2. The *committed* "exactly this id fails, others
pass/skip" regression test is the real guard and is well-specified; the "empirically
confirm" sentence is a one-time author action, not a CI assertion, and reads as redundant
with the pinned purity test. **Suggested fix:** reword so it is clear the pinned regression
test (plus the generator self-assertion) *is* the confirmation — there is no separate manual
acceptance gate a reviewer must trust.

### 5. (low, spec-drift cosmetic) Pre-existing test name `report_lists_all_nineteen_ids...` mislabels 20 ids

Not introduced by MS8, but S1/S6 re-run it. The test `report_lists_all_nineteen_ids...`
(validate.rs:1721) actually asserts 20 ids (its own body comments the "19" is an
off-by-one). Since S6 adds a README §14 matrix and a drift-guard test, this is a natural
place to ensure the docs do not inherit the "19" mislabel. **Suggested fix:** ensure the S6
README matrix states 20 ids; optionally rename the stale test in a test-only S6 touch (still
green, still in scope).
